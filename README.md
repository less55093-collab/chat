# Chat 技能套件

这是一个面向 Codex / oh-my-codex 的本地技能项目，核心目标是把“模糊想法”变成“可执行、可验证、边界清晰的工作计划”，再由明确的执行入口完成实现。

它不是传统意义上的前端或后端应用，也没有运行服务、构建产物或业务代码入口。项目主要由多个 `SKILL.md`、对应的 `agents/openai.yaml` 元数据，以及 `$chat -> $work` 共享交接协议组成，用来定义 Codex 在需求讨论、项目初始化、方案发散和计划执行时的行为边界。

## 核心技能

### `$chat`

`$chat` 是只读的需求发现技能。它适合在目标还不够清晰时使用：Codex 会先检查代码库，理解当前业务结构和约束，再通过多轮少量问题逐步确认需求。

它的重点不是马上写代码，而是形成一个可以交给 `$work` 执行的计划草案。计划中会包含目标、非目标、验收标准、实现步骤、风险和推荐的执行模式。

`$chat` 的重要边界：

- 不修改文件
- 不安装依赖
- 不创建原型
- 不提交或推送代码
- 即使用户说“可以”“开始吧”，也不会自动进入实现
- 只有用户显式调用 `$work` 后，才允许进入编辑和实现阶段

### `$work`

`$work` 用来执行已经确认过的 `$chat` 计划。它不会重新发明一个新方案，而是消费 `$chat` 产出的任务胶囊、验收标准和验证方式。根目录的 `TASK_CAPSULE_SCHEMA.md` 定义了 `$chat` 产出和 `$work` 消费的共享交接字段。

它支持三种执行模式：

- `simple-main`：小而明确的任务，由主代理直接实现和验证，不创建 `WORK_STATE.md`。
- `state-main`：多步骤但不适合委派的任务，由主代理按状态机推进，并维护 `WORK_STATE.md`。
- `delegated-state`：较复杂、可拆分的任务，通过 `WORK_STATE.md` 管理状态，并由子代理执行受限任务，主代理负责最终验证。

`$work` 还支持线性计划和分支计划。分支计划用于处理无法通过只读分析提前解决的不确定实现路径；每个分支都需要明确的成功标准、失败标准和回滚边界。

### `$chat-init`

`$chat-init` 用来初始化新项目或既有项目的规划文档。它通过多角色视角梳理需求，包括产品、架构、UX / 工作流、数据、API / 集成、安全 / 运维、测试、创新和 GitHub 相似项目研究。

它适合在项目刚开始、方向还比较粗糙时使用。输出通常是按项目需要生成或更新的文档，例如：

- `PROJECT_BRIEF.md`
- `PRD.md`
- `ARCHITECTURE.md`
- `ACCEPTANCE_CRITERIA.md`
- `DECISIONS.md`
- `API.md`
- `DATABASE.md`
- `UX_FLOWS.md`
- `TEST_PLAN.md`
- `ROADMAP.md`

`$chat-init` 只在用户确认初始化简报和文档列表后才会创建或更新文档。

### `/explore`

`/explore` 是只读的发散方案技能。它会针对一个问题生成 5 种明显不同的解决方向，并为每种方向给出：

- 核心思路
- 生成概率评分，范围为 1-10
- 稀有度标签：`常见`、`少见` 或 `极度罕见`
- 适用场景
- 主要代价
- 主要风险
- 下一步验证方式

它适合在还没决定路线之前使用，帮助比较常规方案、平衡方案、系统化方案、产品化方案和更少见的高上限方案。

## 典型工作流

1. 使用 `$chat-init` 初始化项目文档，或者使用 `$chat` 讨论一个具体改动。
2. Codex 只读检查项目，并通过多轮问题确认意图、边界和验收标准。
3. `$chat` 按 `TASK_CAPSULE_SCHEMA.md` 输出确认就绪的计划胶囊和任务胶囊，并标注执行模式。
4. 用户显式调用 `$work`。
5. `$work` 消费这些胶囊执行任务，必要时维护 `WORK_STATE.md`。
6. 主代理运行验证、检查 diff，并报告完成证据和剩余风险。

## 项目结构

```text
.
├── SKILL.md                    # $chat 主技能定义
├── TASK_CAPSULE_SCHEMA.md      # $chat -> $work 共享交接协议
├── agents/
│   └── openai.yaml             # $chat 的界面元数据
├── work/
│   ├── SKILL.md                # $work 执行技能定义
│   └── agents/openai.yaml
├── chat-init/
│   ├── SKILL.md                # $chat-init 项目初始化技能定义
│   └── agents/openai.yaml
├── chat-explore/
│   ├── SKILL.md                # /explore 发散方案技能定义
│   └── agents/openai.yaml
└── WORK_STATE.md               # 最近一次 $work 执行状态记录
```

## 设计原则

- 先理解，再计划，再执行。
- 讨论和执行分离，避免把模糊想法直接变成代码改动。
- 读写边界明确，保护用户已有工作。
- `$chat -> $work` 使用共享任务胶囊协议，避免计划和执行端协议漂移。
- 小任务轻量处理，大任务使用状态机和必要的子代理协作。
- 每个任务都要有验收标准和验证方式。
- 分支方案只用于真实不确定性，不用于普通偏好选择。

## 安全边界

这个项目非常强调“什么时候不能动文件”：

- `$chat`、`$chat-init` 的确认前阶段和 `/explore` 都是只读模式。
- `$work` 必须由用户显式调用，才能进入编辑阶段。
- 子代理只能读取和修改任务允许的范围。
- 回滚只能针对当前失败分支的本地改动，不能使用宽泛破坏性的重置命令。
- 主代理必须检查 diff 并完成验证，不能让子代理自己批准自己的工作。

## 适合谁使用

这个技能套件适合希望 Codex 在动手写代码前先充分理解需求的人，尤其适合：

- 想把抽象想法拆成可执行任务的开发者
- 需要代码库上下文辅助需求澄清的项目
- 希望强制区分“讨论”和“实现”的团队
- 需要多步骤执行状态记录和验证闭环的工作流
- 想在实现前比较多种方案的产品或工程任务
