# 更新
因为目前vibecoding 大家重复造轮子的可能性很大，所以我让chat在复杂问题上会优先寻找可复用的一些经验或者工具，再使用这些经验或工具来完成具体任务。

新增和强化了 `$chat-deep`：它不是“上来就搜索”的技能，而是先用多轮问题把用户真正想解决的问题、背景、约束和成功标准问清楚，再做外部深度调研、竞品 / 先例 / 工具 / 库 / API / MCP / 插件发现，最后输出一份可以直接交给 `$work` 执行的研究简报和任务交接。

# Chat 技能套件

这是一个面向 Codex / oh-my-codex 的本地技能项目，核心目标是把“模糊想法”变成“可执行、可验证、边界清晰的工作计划”，再由明确的执行入口完成实现。

它不是传统意义上的前端或后端应用，也没有运行服务、构建产物或业务代码入口。项目主要由多个 `SKILL.md`、对应的 `agents/openai.yaml` 元数据，以及 `$chat / $chat-deep -> $work` 共享交接协议组成，用来定义 Codex 在需求讨论、深度调研、项目初始化、方案发散和计划执行时的行为边界。

## 核心技能

### `$chat`

`$chat` 是只读的需求发现技能。它适合在目标还不够清晰时使用：Codex 会先检查代码库，理解当前业务结构和约束，再通过多轮少量问题逐步确认需求。

它的重点不是马上写代码，而是形成一个可以交给 `$work` 执行的计划草案。计划中会包含目标、非目标、验收标准、实现步骤、风险和推荐的执行模式。

`$chat` 的重要边界：

- 不修改文件
- 不安装依赖
- 不创建原型
- 不提交或推送代码
- 即使用户说“可以”“开始吧”，也不会自动进入实现
- 只有用户显式调用 `$work` 后，才允许进入编辑和实现阶段

`$chat` 的输出语言会跟随用户语言。用户用中文讨论时，理解检查、追问、阶段总结、收尾和计划草案都应该用中文表达；`$chat`、`$work`、`simple-main` 等固定协议词可以保留，但需要用中文解释清楚。

多轮对话发散时，`$chat` 结束前会先做一次收尾判断：确认这轮主线、区分本次核心内容和先放旁边的话题，再判断是否已经能交给 `$work`。因此 `$chat` 不一定总是输出实施计划，它可能以三种形式结束：

- 实现交接：需求、范围和验证方式已经清楚，可以输出计划胶囊和任务胶囊，等待用户显式调用 `$work`。
- 决策记录：这轮只是形成了原则、取舍或方向，不需要立刻实现。
- 继续讨论简报：问题还没聊透，留下当前理解、已确认内容、开放问题和下一轮建议。

### `$chat-deep`

`$chat-deep` 是带外部调研能力的深度需求发现技能。它适合复杂、模糊、成本较高、依赖当前生态信息、需要技术选型、架构比较、竞品 / 先例研究、工具发现、插件 / MCP / API 评估，或者用户明确希望“先研究透再动手”的任务。

它最重要的特点是：先把问题问准，再开始调研。很多用户一开始描述的是表层诉求，真正的目标、上下文、约束和成功标准需要靠追问挖出来。`$chat-deep` 因此会先进入多轮访谈阶段，每轮问少量高价值问题，并在给调研方案前总结一版“问题框定”，让用户确认或修正。

`$chat-deep` 会重点问清：

- 用户真实目标，以及这次调研要支持什么决策。
- 背景、使用者、业务或技术场景，以及为什么现在需要解决。
- 现有代码库、工具、供应商、流程、历史尝试和不能破坏的约束。
- 时间、预算、平台、语言、部署、安全、合规、数据访问、许可证和团队能力。
- 成功标准、失败标准、必须要有、可以没有、明确不做什么。
- 用户已经在考虑的方案、偏好、假设和担心。
- 最终需要的输出形式，例如研究简报、方案比较、架构建议、供应商短名单、风险分析或 `$work` 交接。

确认问题框定后，`$chat-deep` 才会提出调研计划。对工具、集成、自动化、CLI、MCP、插件、API 和开发工作流类任务，它默认采用 reuse-first：先寻找已经存在的开源项目、成熟工具、官方 API、SaaS、模板、协议、行业实践和可参考实现，只有在这些方案不能满足核心约束时，才建议自研。

它会把调研过程沉淀为可恢复的 markdown 状态，通常写在 `.codex/research/YYYY-MM-DD-topic/`：

- `RESEARCH_STATE.md`：当前状态、范围、下一步和防重复记录。
- `SOURCES.md`：采纳和拒绝过的来源、项目、工具和理由。
- `FINDINGS.md`：提炼出的事实、判断、证据和不确定性。
- `DECISIONS.md`：推荐方向、拒绝方案、延期问题和依据。
- `RESEARCH_BRIEF.md`：最终给用户和 `$work` 使用的研究简报。

`$chat-deep` 的边界也很明确：它可以在用户确认后安装或配置调研工具，可以写研究文档，但不修改业务代码、不提交、不部署，也不会把“同意调研方案”理解成“开始实现”。真正实现仍然需要用户显式调用 `$work`。

### `$work`

`$work` 用来执行已经确认过的 `$chat` 计划。它不会重新发明一个新方案，而是消费 `$chat` 产出的任务胶囊、验收标准和验证方式。根目录的 `TASK_CAPSULE_SCHEMA.md` 定义了 `$chat` 产出和 `$work` 消费的共享交接字段。

它支持三种执行模式：

- `simple-main`：小而明确的任务，由主代理直接实现和验证，不创建 `WORK_STATE.md`。
- `state-main`：多步骤但不适合委派的任务，由主代理按状态机推进，并维护 `WORK_STATE.md`。
- `delegated-state`：较复杂、可拆分的任务，通过 `WORK_STATE.md` 管理状态，并由子代理执行受限任务，主代理负责最终验证。

`$work` 还支持线性计划和分支计划。分支计划用于处理无法通过只读分析提前解决的不确定实现路径；每个分支都需要明确的成功标准、失败标准和回滚边界。

### `$chat-init`

`$chat-init` 用来初始化新项目或既有项目的规划文档。它通过多角色视角梳理需求，包括产品、架构、UX / 工作流、数据、API / 集成、安全 / 运维、测试、创新和 GitHub 相似项目研究。

它适合在项目刚开始、方向还比较粗糙时使用。输出通常是按项目需要生成或更新的文档，例如：

- `PROJECT_BRIEF.md`
- `PRD.md`
- `ARCHITECTURE.md`
- `ACCEPTANCE_CRITERIA.md`
- `DECISIONS.md`
- `API.md`
- `DATABASE.md`
- `UX_FLOWS.md`
- `TEST_PLAN.md`
- `ROADMAP.md`

`$chat-init` 只在用户确认初始化简报和文档列表后才会创建或更新文档。

### `/explore`

`/explore` 是只读的发散方案技能。它会针对一个问题生成 5 种明显不同的解决方向，并为每种方向给出：

- 核心思路
- 生成概率评分，范围为 1-10
- 稀有度标签：`常见`、`少见` 或 `极度罕见`
- 适用场景
- 主要代价
- 主要风险
- 下一步验证方式

它适合在还没决定路线之前使用，帮助比较常规方案、平衡方案、系统化方案、产品化方案和更少见的高上限方案。

## 典型工作流

1. 使用 `$chat-init` 初始化项目文档，或者使用 `$chat` 讨论一个具体改动。
2. Codex 只读检查项目，并通过多轮问题确认意图、边界和验收标准。
3. `$chat` 按 `TASK_CAPSULE_SCHEMA.md` 输出确认就绪的计划胶囊和任务胶囊，并标注执行模式。
4. 用户显式调用 `$work`。
5. `$work` 消费这些胶囊执行任务，必要时维护 `WORK_STATE.md`。
6. 主代理运行验证、检查 diff，并报告完成证据和剩余风险。

## 深度调研工作流

1. 用户调用 `$chat-deep`，提出一个复杂、模糊或需要外部信息支撑的问题。
2. `$chat-deep` 先多轮访谈，不急着给调研方案，直到真实目标、背景、约束和成功标准足够清楚。
3. `$chat-deep` 输出问题框定，让用户确认或修正。
4. `$chat-deep` 基于确认后的问题框定提出调研深度、来源范围、候选工具和研究文档路径。
5. 用户确认调研计划后，`$chat-deep` 创建或恢复 `.codex/research/...` 状态文件。
6. `$chat-deep` 执行外部调研、reuse-first 候选发现、证据提取、方案比较和风险验证。
7. `$chat-deep` 输出 `RESEARCH_BRIEF.md`，其中包含推荐方向、为什么不是其他方案、验证策略和 `$work` 可执行交接。
8. 用户显式调用 `$work` 后，才进入代码实现或项目变更。

## 设计原则

- 先理解，再计划，再执行。
- 深度调研前先问准问题，避免调研一个用户并不真正需要的问题。
- 讨论和执行分离，避免把模糊想法直接变成代码改动。
- 复杂问题默认 reuse-first，优先找已有工具、库、API、插件、协议、模板和行业实践。
- 读写边界明确，保护用户已有工作。
- `$chat / $chat-deep -> $work` 使用共享任务胶囊协议，避免计划和执行端协议漂移。
- 小任务轻量处理，大任务使用状态机和必要的子代理协作。
- 每个任务都要有验收标准和验证方式。
- 分支方案只用于真实不确定性，不用于普通偏好选择。

## 安全边界

这个项目非常强调“什么时候不能动文件”：

- `$chat`、`$chat-init` 的确认前阶段和 `/explore` 都是只读模式。
- `$chat-deep` 可以写研究文档和状态文件，但不能改业务代码，不能把调研确认当成实现确认。
- `$work` 必须由用户显式调用，才能进入编辑阶段。
- 子代理只能读取和修改任务允许的范围。
- 回滚只能针对当前失败分支的本地改动，不能使用宽泛破坏性的重置命令。
- 主代理必须检查 diff 并完成验证，不能让子代理自己批准自己的工作。

## 适合谁使用

这个技能套件适合希望 Codex 在动手写代码前先充分理解需求的人，尤其适合：

- 想把抽象想法拆成可执行任务的开发者
- 需要代码库上下文辅助需求澄清的项目
- 需要先把复杂问题问清楚，再做外部调研和方案比较的团队
- 想减少重复造轮子，优先复用成熟工具、库、API、MCP 或插件的开发者
- 希望强制区分“讨论”和“实现”的团队
- 需要多步骤执行状态记录和验证闭环的工作流
- 想在实现前比较多种方案的产品或工程任务
